# PM Simulator ゲームデザイン仕様書 (星レベル・情報隠蔽対応版)

本ドキュメントは、「PM Simulator」のゲームルール、各種エンティティのデータ構造、およびコアループ内の計算モデルを定義します。

---

## 🎯 ゲームの基本目的と勝利条件
プレイヤーの目標は、自分をプロジェクトにアサインした**「上級マネージャー（上司）」を満足させること**です。
上級マネージャーの評価は、プロジェクトの**「最終的な顧客満足度」**、PMとしての**「適切な管理（報告・エスカレーション）」**、そして**「チームの健全性（過労離職者を出さない）」**によって決定されます。

---

## 🔁 ゲームサイクル（週次スプリント進行）

ゲームは「1週間（5営業日 / 1スプリント）」を最小の進行単位とし、キックオフから納品までをシミュレートします。

```
 [1. キックオフフェーズ] 
     - 引き合い（星評価） ➔ 体制構築（雇用） ➔ 詳細ヒアリング (PL同行) ➔ 見積もり ➔ 事前交渉
            │
            ▼
 [2. スプリント開始（PMの方針・ケア指示）] 
            │
            ▼
 [3. 1週間の自動シミュレーション（PLによる自動アサインと開発。PMはセリフで状態を察する）]
            │
            ▼
 [4. スプリント終了（週次ダッシュボード確認）]
            │
            ▼
 [5. 週次トラブル・イベントへの対応 ＆ 顧客・上司交渉]
            │
            ▼ (クリアまたは破綻するまで 2-5 を繰り返す)
```

## 🤝 1. キックオフフェーズ（立ち上げ時の防衛と交渉）

プロジェクト開始前に、PMは初期の不確実性と過酷な前提条件を解消するため、4つのステップを経てプロジェクトの初期状態を構築・調整します。

### STEP 1: 初期引き合い（星パラメータによるレベル感の提示）
顧客からプロジェクトの相談（引き合い）が届きます。PMにはプロジェクトの前提レベル感が **5段階（🌟1〜5）** で開示されます。

* **要求具体度（Clarity Level）**: 顧客の要望がどれだけ固まっているか。星が低いほど、開始後に「仕様変更・手戻り（追加タスク）」が高確率で発生する。
* **予算妥当性（Budget Level）**: 予算が適正か。星が低いほど初期予算が少なく、人件費だけでカツカツになる。
* **納期妥当性（Schedule Level）**: スケジュールが適正か。星が低いほど初期納期が短く、通常稼働だけでは間に合わない。

### STEP 2: 体制構築（要員アサイン）
提示されたレベル感を見て、雇用するPLおよびDEVを選択します。
* **PL（プロジェクトリーダー）の専門知識**:
  * **ケンPL (ベテラン / サーバー側知識豊富)**: サーバー側（ロジック）開発に高い知識を持つ。日当 ¥18,000。
  * **レンPL (若手 / 画面側知識豊富)**: 画面側（UI）開発に高い知識を持つ。日当 ¥10,000。
* **DEV（開発メンバー）の専門領域**:
  * **タクDEV (サーバー側特化/日当 ¥15,000)**: サーバー側タスクの生産速度 1.5倍。画面側タスクは苦手（速度 0.6倍、バグ率 3倍）。
  * **ユイDEV (画面側特化/日当 ¥12,000)**: 画面側タスクの生産速度 1.2倍。サーバー側タスクは苦手（速度 0.5倍、バグ率 3倍）。

### STEP 3: 詳細ヒアリング（PL同行要件定義）
雇用したPLを同行させて、詳細なヒアリング（要件定義ミーティング）に臨みます。
1. **丁寧なヒアリング (PLを同行させる / 納期を1週消費)**
   * 納期を1週消費するため、**納期妥当性レベルが -1 ダウン**する。
   * **プロジェクトの主特性とPLの専門知識が一致している場合**:
     * PLが要望を的確に整理できるため、**要求具体度レベルが +3 アップ（最大5）**、初期顧客満足度 +15。
   * **一致していない場合（要件定義の罠）**:
     * 専門知識が噛み合わず議論が難航し、時間だけを浪費して**要求具体度レベルは +1 アップ**に留まる。初期満足度 +5。
2. **簡易ヒアリング (PMが一人で行う / 即日キックオフ)**
   * 技術的な突っ込みができず、星変動なし。
3. **ヒアリングなし (即日キックオフ、納期+1週間)**
   * 納期に1週間の猶予が生まれるため、**納期妥当性レベルが +1 アップ**する。
   * ただし顧客を軽視したため初期満足度が低下（-15）、確認不足で**要求具体度レベルが -1 ダウン（最小1）**、開発時の初期バグ混入率が 1.5倍 になる。

### STEP 4: PL見積もり ➔ 事前交渉
詳細ヒアリングを終えた段階で、PLから見積もり妥当性の監査レポート（エビデンス）が提出されます。このレポートを元に、初期の納期・予算・スコープ交渉を行います。

* **納期延長交渉**: 納期を+1週間し、**納期妥当性レベルを +1 アップ**する。
* **予算追加交渉**: 予算を+¥300,000し、**予算妥当性レベルを +1 アップ**する。
* **スコープ削減交渉（機能を削る）**: 開発する機能の一部（TODOタスクの最後から2個）をあらかじめ削減し、作業量を減らすことで、**納期妥当性と予算妥当性レベルを同時に +1 アップ**する。ただし、機能を諦めさせるため顧客満足度が低下する。

> [!NOTE]
> 交渉はPLの見積もりレポート（エビデンス）を提示して行う場合、納得感があるため顧客満足度・上司評価への影響を最小限（-5%）に抑えられます。エビデンスなしで交渉を強行した場合は、説得力がないため大幅なペナルティ（-25%）が発生します。

## 📊 2. 週次スプリント監視と計算モデル

### 1. 情報隠蔽化（パラメータの非表示）とセリフによる察知
PMとしての「現場に潜む人間的な状態を察する」体験を高めるため、画面上からシステム的な表示を徹底的に隠蔽します。
* メンバーの `[疲労度]` `[士気]` `[良好/要注意/限界]` などのステータス表示は**完全に非表示（非公開）**となります。
* タスクボードの `[※スキルミスマッチ中]` などのアシスト表示も非表示となります。
* メンバーは、自身の状態（ミスマッチでのアサイン、疲労の蓄積、限界寸前）に応じて、発言（セリフ）のバリエーションを変化させます。PMはメンバーの**言葉（発言）**と、**タスクの消化速度**からチームの状態を推測・察知し、手を打つ必要があります。

### 2. 週次トラブル（仕様変更）と要求具体度
毎週末の「追加要望（手戻り）」の発生確率は、キックオフフェーズでどれだけ要件定義を詰めたか（**要求具体度レベル**）に依存して決定されます。
* 要求具体度 🌟5 ➔ 仕様変更発生率: 5% (極めて安定)
* 要求具体度 🌟4 ➔ 仕様変更発生率: 15%
* 要求具体度 🌟3 ➔ 仕様変更発生率: 30%
* 要求具体度 🌟2 ➔ 仕様変更発生率: 50%
* 要求具体度 🌟1 ➔ 仕様変更発生率: 80% (手戻りが毎週のように多発)

## 🧠 3. PMの週次戦略アクション

毎週の開始時、PMは以下のマクロコマンドを実行できます。

* **PLへの開発方針指示**: 通常（進捗優先）または バグ修正優先を指示する。
* **PLを介した要員指示 (残業・休暇)**:
  * PLを通じて「来週、タクを休ませてくれ」「来週は残業でカバーしてくれ」と指示します。
  * 疲労が高いメンバーに残業を強行させようとするとPLから進言されますが、これを無視して強行すると、PLのプライドが傷つき**PL士気が大幅に低下（過干渉ペナルティ: -25）**します。PL士気が0になるとボイコット（自動アサイン停止）が発生します。
* **PLとのスプリント面談（1on1）**: 
  * 1on1を行っても数値パラメータは開示されません。代わりに、PLからは「メンバーの健康状態（言葉による定性報告）」、DEVメンバーからは「本音のコメント（残業への不満や休暇要望）」を聞き出し、間接的にチームの状態を正確に把握します。
* **顧客との週次定例（デモ報告）**: 進捗報告とデモを行います。要求具体度レベルを向上させ、将来の仕様変更率を下げます。
* **上司へのエスカレーション交渉**: トラブル発生時に追加予算や納期延長をエスカレーションします（上司信頼度を消費）。
* **PLへのケア（謝罪）**: PLがボイコットした際、予算を ¥50,000 消費して関係を修復し、自動アサインを復旧させます。
