# PM Simulator ゲームデザイン仕様書 (マクロPM視点版)

本ドキュメントは、「PM Simulator」のゲームルール、各種エンティティのデータ構造、およびコアループ内の計算モデルを定義します。

---

## 🎯 ゲームの基本目的と勝利条件
プレイヤーの目標は、自分をプロジェクトにアサインした**「上級マネージャー（上司）」を満足させること**です。
上級マネージャーの評価は、プロジェクトの**「最終的な顧客満足度」**、PMとしての**「適切な管理（報告・交渉）」**、そして**「チームの健全性（過労離職者を出さない）」**によって決定されます。

---

## 🔁 ゲームサイクル（週次スプリント進行）

ゲームは「1日」ではなく**「1週間（5営業日 / 1スプリント）」**を最小の進行単位とします。日々の細かいアサインや進捗管理はPL（プロジェクトリーダー）に一任し、PMはスプリント単位での監視と、節目での重要な意思決定に専念します。

```
 [1. キックオフ（事前交渉・雇用）] （初回のみ）
         │
         ▼
 [2. スプリント開始（PMの方針・ケア指示）] 
         │
         ▼
 [3. 1週間の自動シミュレーション（PLによる自動アサインと開発）]
         │
         ▼
 [4. スプリント終了（週次ダッシュボード確認）]
         │
         ▼
 [5. 週次トラブル・イベントへの対応 ＆ 顧客・上司交渉]
         │
         ▼ (クリアまたは破綻するまで 2-5 を繰り返す)
```

---

## 🤝 1. キックオフフェーズ（事前交渉と体制構築）

プロジェクト開始前に、PMは以下の条件設定と交渉を行います。

### A. 初期契約交渉（トレードオフ）
以下の3つから1つを選択し、プロジェクトの初期条件を決定します。
1. **納期延長交渉**: 納期を +1週間（5日）するが、顧客満足度の初期値が下がる（-20）。
2. **予算追加交渉**: 予算を +¥300,000 増やすが、上級マネージャーの初期信頼度が下がる（-20）。
3. **現状維持**: 初期値（納期4週間/20日、予算¥1,000,000、顧客満足度80%、上司信頼度80%）で開始。

### B. 体制構築（要員アサイン）
獲得した予算の範囲内で、雇用するメンバーを決定します。
* **PL（プロジェクトリーダー）の選択 (必須・1名)**:
  * **鈴木PL (ベテラン)**: 日当 ¥18,000。優秀で自動アサイン効率が高く、バグ発生率を抑える。管理者プライドが高く、PMから直接DEVへ過干渉されると士気が下がりやすいが、ボイコットからの復帰が早い。
  * **田中PL (若手)**: 日当 ¥10,000。アサイン能力がやや低く、バグ発生率を抑えられない。気弱で、過干渉やトラブルが発生するとすぐに心が折れてボイコットする。
* **DEV（開発者）の選択 (1〜2名)**:
  * **山田 (中堅)**: 日当 ¥15,000。作業速度が早く、バグ率中。飲み会を好む。
  * **佐藤 (若手)**: 日当 ¥12,000。作業速度は遅いが、バグ率が極めて低い。プライベート重視（残業や飲み会強制を嫌う）。

## 📊 2. 週次スプリント監視と計算モデル

1週間（5営業日）が進む際、以下の処理が自動シミュレーションされます。

### 1. 給与の消費
* 雇用しているPLおよびDEV全員の5日分の給与がプロジェクト予算から引かれます。

### 2. PLによる自動アサインと開発
* PLは、DEVがIdle（空き）状態になると、TODOタスクから自動的にタスクをアサインします。
  * PMがPLに「バグ修正優先」を指示している場合、バグが1件以上あれば優先して「バグ修正タスク」を割り当てます。
  * 田中PLの場合、アサインにミスが発生し、1日待機状態になることがあります。

### 3. 進捗とバグの計算
* DEVはアサインされたタスクを進めます。
  * 1週間の進捗（%） = 開発者の作業速度 * 補正係数 * 稼働時間 (通常40時間)
  * 補正係数 = (士気と疲労による影響)
  * バグ発生確率: 疲労が高まるとバグ発生率が最大で2倍に増加。
* 鈴木PLの場合、DEVのバグ発生率を 10% 低減させるパッシブ効果を持ちます。

### 4. 疲労と士気の自動推移（隠しパラメータ）
* 1週間稼働したDEVは、疲労が蓄積し、士気が少し低下します。
* PMは週次のダッシュボードで、メンバーの正確な数値ではなく、PLがまとめた「良好 / 要注意 / 限界」などの大まかなサインで推移を確認します。

## 🛡 3. トラブルイベントとPMの防波堤機能

週の終わりに、顧客の「要求あいまい度」や進行状況に応じてトラブルイベント（仕様変更要求など）が発生します。
特に追加要望のイベントでは、PMの「開発者を守る防波堤（クッション）としての役割」が試されます。

1. **丸投げする**
   - **結果**: 顧客満足度は大きく上がるが、担当DEVの士気が大暴落し、過労離職のリスクが激増する。
2. **防波堤として調整する (APまたは予算の消費)**
   - **結果**: PMがクッションとなり、納得感を持って依頼する。追加の調整コスト（予算 ¥30,000）が発生するが、開発者の士気低下を最小限に抑える。
3. **顧客と交渉して断る**
   - **結果**: 追加タスクは発生しないが、顧客満足度が大幅に低下する。

## 🧠 4. PMの週次戦略アクション

毎週の開始時、PMは以下のマクロコマンドを実行できます。

* **PLへの開発方針指示**: 通常（進捗優先）または バグ修正優先を指示する。
* **PLを介した要員指示 (残業・休暇)**:
  * 直接DEVに命令するのではなく、PLを通じて「来週、山田を休ませてくれ」「来週は残業で挽回してくれ」と指示します。
  * PLから「これ以上の残業はメンバーが倒れます」と進言されたのを無視して無理をさせると、PLおよびDEVの士気が下がります。
  * 直接DEVに無理な指示を通すと、**PLのプライドが傷つきPL士気が大幅に低下（過干渉ペナルティ）**します。PL士気が0になるとボイコット（自動アサイン停止）が発生します。
* **PLとのスプリント面談**:
  * PLと面談を行い、PLの士気を回復させます。また、PLの信頼を得ることで、メンバー全員の正確な疲労・士気数値をレポートとして開示させます。
* **顧客との週次定例（デモ報告）**:
  * 進捗報告とデモを行います。顧客のタイプに合わせて期待値調整を行い、顧客の「要求あいまい度」を 20% 減少させ、将来の仕様変更率を下げます。
* **上司へのエスカレーション交渉**:
  * 予算不足や納期遅延のリスクに対し、上司に予算の追加や納期延長の根回しを交渉します（上司信頼度を消費）。
* **PLへのケア（謝罪）**:
  * PLがボイコットした際、予算を ¥50,000 消費して関係を修復し、自動アサインを復旧させます。
