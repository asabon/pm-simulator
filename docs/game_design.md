# PM Simulator ゲームデザイン仕様書

本ドキュメントは、「PM Simulator」の全体コンセプト、ライフサイクル、コアループ、および主要なシミュレーションメカニクスを定義するハイレベル設計書です。

---

## 🎯 1. ゲーム概要とコアコンセプト

「PM Simulator」は、プレイヤーがシステム開発プロジェクトの**プロジェクトマネージャー（PM）**となり、長年にわたりプロジェクトやチームを成功へと導く**通年キャリアマネジメント・シミュレーション**です。

### 基本理念
* **「万能の解決策はない」**: 単にメンバーを詰めたり、会議を増やしたりしても問題は解決しません。すべてのアクションにはメリットと副反応（トレードオフ）が存在します。
* **「人は数値ではない」**: チームメンバーは年齢、感情、疲労、得意分野などの個性を持ちます。初期状態では彼らの本音や真の能力は見えず、長く協働することで徐々に解像度が上がっていきます。
* **「現場は自走し、PMは方針を示す」**: 実際の開発タスクのアサインは現場のPL（プロジェクトリーダー）に委任され、時間経過とともに自動で進行します。PMは現場に介入しすぎず、適切な「方針提示」と「環境調整・リスク予防」で成功率を高めます。

### 全体構造（キャリア / 通年経営型）
本ゲームは単発のプロジェクトクリアで終わるのではなく、**PMとしてのキャリアを重ねながら、複数プロジェクトを継続的に回していく構造**をとります。
* メンバーとの信頼関係やパラメータの解像度（親密度・理解度）は、プロジェクトを跨いで継続・引き継がれます。
* メンバーの加齢や成長、定年退職、新入社員の配属など、中長期的な組織マネジメントの要素も含みます。

### 勝利・評価条件
PMの評価は、各プロジェクトの成果およびキャリア通算で以下の **3つの軸** の総合バランスで決定されます。
1. **顧客満足度**: 納期、予算、品質、および顧客との適切なコミュニケーション。
2. **上司評価（社内信頼）**: 適切な報告・事前交渉・エスカレーションが行われているか。
3. **チームの健全性**: メンバーの過労離職、ボイコット、極度なモチベーション低下を出さないこと。

## 🚩 2. プロジェクトの全体フェーズ構成 (Project Life Cycle)

1つのプロジェクトは、大きく以下の **3つのフェーズ** およびプロジェクト間の **人事・組織更新** で進行します。

```
 [1. プロジェクトキックオフフェーズ]
      ・上司からの案件アサイン ＆ 期待値提示受領
      ・前提確認・体制選択（人選）・事前交渉を行って防衛ラインを張る準備段階
            │
            ▼
 [2. プロジェクト進行フェーズ] (メインゲームプレイ / イベント駆動型)
      ・時間の自動経過中、定期イベント（定例）および突発緊急イベントが発生し、即座の対応・介入を行う
            │
            ▼
 [3. プロジェクトクロージング ＆ 人事・組織更新フェーズ]
      ・成果物の納品、プレイスコア確定
      ・【人事・組織更新】メンバーの加齢/成長、定年退職者の発生、新入社員の配属、解像度引き継ぎ
```

### 1. プロジェクトキックオフフェーズ
* **目的**: 上級マネージャーから案件の打診・アサインを受け、プロジェクト開始前の初期条件（予算・納期・要求の曖昧さ・上司の期待）を確認し、防衛ラインを構築する。
* **プレイヤーのアクション ＆ 発生フロー**:
  * **ステップ1: 上級マネージャーからの案件アサイン ＆ 期待値提示**
    * 上級マネージャーから「〇〇さん、今回の案件のPMをお願いしたい」と案件の打診を受け取る。
    * 案件の概略・初期難易度レベル（🌟1〜5）とともに、**今回上司が最も重視する期待項目**（例：納期厳守 / 予算遵守 / 顧客満足 / 新規顧客の信頼獲得など）が提示される。
  * **ステップ2: 体制選定（メンバープールからの人選）**
    * 現在所属しているチームメンバープールから、今回のプロジェクト要件（バックエンド/フロントエンド等）に適したPLおよびDEVメンバーを選定・配置する。
  * **ステップ3: 前提確認と事前交渉（防衛ラインの構築）**
    * 詳細ヒアリングや顧客・上司との事前交渉（予算追加・納期調整・機能削り）を行い、無謀な初期条件に対する防衛線を張る。

### 2. プロジェクト進行フェーズ (イベント駆動型)
* **目的**: 時間の自動進行中、発生するイベント（定期報告・突発トラブル）に応じて適切な介入を行い、安全かつ確実に進める。
* **主要なゲームプレイ**: 後述する「イベント駆動型コアループ」に基づき、定期報告と緊急割り込みアラートへ対応（AP消費）する。

### 3. プロジェクトクロージング ＆ 人事・組織更新フェーズ
* **目的**: プロジェクトの締めくくりと評価確定、およびプロジェクト間の**人事・組織更新（年度更新）**。
* **プレイヤーのアクション ＆ 発生イベント**:
  * **評価精算**: 成果物の納品と最終結果の評価（顧客満足度・社内評価・チーム健全性スコアの確定）。
  * **成長と加齢**: 参加メンバーの経験値加算、解像度の向上、および年齢＋1。
  * **人事イベント（組織変動）**:
    * **定年退職**: 長年貢献してくれた頼れるベテラン社員の退職。
    * **新入社員の配属**: 知識ゼロ・パラメータ完全未知（初期状態）だが、若くて残業耐性が高い新人がチームに配属。
    * **異動・採用**: 組織の増減やメンバー入れ替え。
  * **次案件への引き継ぎ**: 変化した新たなチーム体制で、次のプロジェクトのキックオフ（フェーズ1）へと向かう。

## 🔁 3. イベント駆動型コアループ (Event-Driven Loop)

第2フェーズ（プロジェクト進行フェーズ）は、固定のターン制ではなく**「イベント駆動型 (Event-Driven)」**で進行します。時間が自動的に進む中、2種類のイベントが発生したタイミングでPMに対応（AP消費）が求められます。

```
                    【時間の自動進行】
                           │
         ┌─────────────────┴─────────────────┐
         ▼                                   ▼
  【① 定期・スケジュールイベント】     【② 突発・緊急トラブルイベント】
   ・PMが設定した周期（日次/週次/月次）  ・重大バグの発生、キーマンの限界/過労、
     で「必ず発生」する定例ミーティング   ・顧客からの緊急割り込み要求など
   ・報告確認、セリフ対話、方針調整      ・時間が一時停止し、即座の対応を迫られる
```

### 発生する2種類のイベント

1. **① 定期・スケジュール型イベント (Scheduled Events)**
   * **発生タイミング**: PMが設定した監視・報告周期（日次/週次/月次など）の節目で**必ず発生**。
   * **内容**: 定期進捗報告の確認、PLとのセリフ・1on1対話、マクロ方針（ポリシー）の調整、定例頻度の変更。
2. **② 突発・割り込み型イベント (Urgent / Interrupt Events)**
   * **発生タイミング**: 開発中の潜在リスク（バグ数、メンバー疲労、不満、顧客不満など）が一定の閾値を超えた際、**定例の周期に関わらず随時割り込み発生**。
   * **内容**: 重大バグの混入、キーマン（DEV）の限界・体調不良、顧客からの突然の仕様変更・割り込み、PLからの急なエスカレーション。
   * **特徴**: 時間の進行がストップし、PMは即座に限られたAPを消費して対応（方針修正・顧客交渉・上司エスカレーション・メンタルケア等）を迫られる。

## ⚙️ 4. 主要メカニクスとシミュレーション原則

### 4.1 アサインの委任とPM方針提示 (PL Autonomy & PM Policy)
* **アサインの主導権**: メンバー個々のタスク割り当てや日々の開発進捗コントロールは、**原則としてPL（プロジェクトリーダー）に一任**されます。
* **PMの指針（ポリシー）提示**: PMはミクロなアサインではなく、「今期は納期優先でバグは後回しにしてほしい」「品質重視でテストを手厚くしてほしい」といったマクロな方針提示や外圧からの防衛を行います。

### 4.2 PMのアクションコストとタイムマネジメント (Action Cost)
* **AP制限**: 定期イベントおよび突発イベント発生時にPMが行動できる量（AP）には上限があります。
* **選択と集中**: 「今発生した緊急トラブル対応にAPを使うか」「定期面談でのチームケアにAPを使うか」という、タイムマネジメントの意思決定を求心力とします。

### 4.3 トレードオフと副作用・ジレンマ (Trade-offs & Side Effects)
すべてのアクションおよび設定変更には必ず**代償（コスト）と副作用（デメリット）**が存在します。

| アクション / 設定 | メリット (期待効果) | デメリット / 副作用 (リスク) |
| :--- | :--- | :--- |
| **日次 (Daily) に監視頻度を上げる** | 毎日超早期にリスク（バグ・疲労）を発見・対応できる | 毎日の報告資料作成やミーティングに追われ**PLの実務が著しく停滞**する。監視感により**PLの不満・ストレスが急増**する |
| **週次 (Weekly) 監視** *(標準)* | 状況把握の精度と現場の実務時間のバランスが良い | 標準的なリスク発生確率と進捗ペースで進行する |
| **月次 (Monthly) に監視頻度を下げる** | 報告のオーバーヘッドが最小限になり、**PL・現場が開発に100%集中できる** | 放置期間が長いため、結果を受け取った時点で**進捗の大幅遅延や大炎上・チーム崩壊が顕在化**しているリスクが高い |
| **要員に残業を指示する** | 短期的な進捗・生産量が大幅に向上する | メンバーの疲労が蓄積し、**バグ混入率が上昇**。長期化すると**離職・ボイコット**が発生する |
| **顧客との詳細ヒアリング** | 要件が明確化し、将来の仕様変更や手戻り率が激減する | PMのAPおよびヒアリング工数を奪われ、**初期納期を消費する** |
| **顧客の要望をすべて受け入れる** | 一時的な顧客満足度が上昇する | スコープが膨れ上がり、**予算オーバーやチームの疲弊**を招く |
| **PLへアサイン指示を強行する** | 作業ミスやスキルのミスマッチを直接補正できる | PLの主体性とプライドを傷つけ、**PL士気が低下（ボイコット）**する |

### 4.4 チームメンバー（「人間」モデル）と人事・解像度メカニクス
メンバーは固定の数値コマではなく、成長・退職・個性を備えた「人間」オブジェクトとして表現されます。

* **年齢・経験属性と世代交代**:
  * **若手・新入社員**: 経験ゼロ・初期解像度は「完全未知」。成長速度が速く残業耐性も高いが、ミス（バグ）を出しやすい。ベテランの元で育てて一本立ちさせる必要がある。
  * **ベテラン社員**: 高品質・即戦力。しかし定年退職のリスクを常に抱えており、退職前に後継者を育成・引き継ぎさせるマネジメントが求められる。無茶な残業指示には強く反発する。
* **パラメータ可視度の段階的解放 (Progressive Revelation)**:
  * **初期（未知）**: 定性的な発言（セリフ）や消化速度の違和感から推測する（特に配属直後の新人は完全未知）。
  * **中期（3段階表示）**: 協働や1on1を経て「高/中/低」「良好/注意/限界」といった粗い粒度で可視化。
  * **熟練（10段階/具体値）**: 長期的な信頼関係により「8/10」「疲労度 72/100」など精緻に把握可能。

## 📝 5. 今後の拡張・詳細化方針

本ハイレベル設計に基づき、プロトタイプ開発（Python）および本開発に向けて以下を順次ブレイクダウンしていきます。
* **データ構造設計**: `PM`（AP制限・サイクル設定）、`Human`（年齢・疲労・解像度・退職フラグ）、`Project`、`Task`、`Event` クラス設計。
* **イベント発生条件ロジック**: スケジュールイベントの発生周期計算、突発イベントの閾値判定。
* **フェーズごとの画面・UIプロトタイプ**: キックオフ画面、イベント割り込み時の緊急ダイアログ画面、クロージング＆人事更新画面。
