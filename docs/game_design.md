# PM Simulator ゲームデザイン仕様書 (要件ヒアリング・アサイン最適化対応版)

本ドキュメントは、「PM Simulator」のゲームルール、各種エンティティのデータ構造、およびコアループ内の計算モデルを定義します。

---

## 🎯 ゲームの基本目的と勝利条件
プレイヤーの目標は、自分をプロジェクトにアサインした**「上級マネージャー（上司）」を満足させること**です。
上級マネージャーの評価は、プロジェクトの**「最終的な顧客満足度」**、PMとしての**「適切な管理（報告・エスカレーション）」**、そして**「チームの健全性（過労離職者を出さない）」**によって決定されます。

---

## 🔁 ゲームサイクル（週次スプリント進行）

ゲームは「1週間（5営業日 / 1スプリント）」を最小の進行単位とし、キックオフから納品までをシミュレートします。

```
 [1. キックオフフェーズ] 
     - 引き合い確認 ➔ 体制構築 ➔ 詳細ヒアリング (PL同行) ➔ 見積もり ➔ 事前交渉
            │
            ▼
 [2. スプリント開始（PMの方針・ケア指示）] 
            │
            ▼
 [3. 1週間の自動シミュレーション（PLによる自動アサインと開発）]
            │
            ▼
 [4. スプリント終了（週次ダッシュボード確認）]
            │
            ▼
 [5. 週次トラブル・イベントへの対応 ＆ 顧客・上司交渉]
            │
            ▼ (クリアまたは破綻するまで 2-5 を繰り返す)
```

---

## 🤝 1. キックオフフェーズ（アサイン最適化とエビデンス交渉）

プロジェクト開始前に、PMは以下の4つのステップを経てプロジェクトの初期状態を構築・調整します。

### STEP 1: 初期引き合い（粗い要求の開示）
顧客（渡辺部長）から「何が作りたいか」の大まかな要望と、プロジェクトの主要なタスク特性の比率（BEが多いプロジェクトか、FEが多いプロジェクトか）がPMに開示されます。

### STEP 2: 体制構築（要員アサイン）
開示されたタスク比率を見て、雇用するPLおよびDEVを選択します。
* **PL（プロジェクトリーダー）の専門知識**:
  * **ケンPL (ベテラン / BE知識豊富)**: バックエンドの多い開発に高い知識を持つ（BE比率の高いプロジェクトのヒアリングで大活躍）。日当 ¥18,000。
  * **レンPL (若手 / FE知識豊富)**: フロントエンドの多い開発に高い知識を持つ（FE比率の高いプロジェクトのヒアリングで大活躍）。日当 ¥10,000。
* **DEV（開発メンバー）の専門領域**:
  * **タクDEV (BE特化/日当 ¥15,000)**: BEタスク速度 1.5倍。FEタスクは苦手（速度 0.6倍、バグ率 3倍）。
  * **ユイDEV (FE特化/日当 ¥12,000)**: FEタスク速度 1.2倍。BEタスクは苦手（速度 0.5倍、バグ率 3倍）。

### STEP 3: 詳細ヒアリング（PL同行要件定義）
雇用したPLを同行させて、詳細なヒアリング（要件定義ミーティング）に臨みます。
1. **丁寧なヒアリング (PLを同行させる / 納期を1週消費)**
   * **プロジェクトの特性とPLの専門知識が一致している場合** (例: BE中心プロジェクトにケンPLが同行):
     * PLが仕様を綺麗に整理し、言語化できない顧客（渡辺部長）からも正確な要求を引き出せます（初期顧客満足度 +15、要求あいまい度 `vague_level` が大幅低下 -45）。
   * **プロジェクトの特性とPLの専門知識がズレている場合** (例: BE中心プロジェクトにレンPLが同行):
     * 専門知識が噛み合わず、有益な要件整理ができません（初期満足度変化なし、`vague_level` は -15 のみ低下。時間を浪費するだけの「要件定義の罠」となります）。
2. **簡易ヒアリング (PMが一人で行う / 即日キックオフ)**
   * 技術的な突っ込みができず、顧客のあいまさに引きずられます（`vague_level` -5% のみ低下）。
3. **ヒアリングなし (即日キックオフ、納期+1週間)**
   * 納期を1週間余分に稼げますが、顧客を軽視したため初期満足度が低下（-15）、`vague_level` +30、初期バグ率 1.5倍。

### STEP 4: PL見積もり ➔ 事前交渉
詳細ヒアリングを終えた段階で、PLから「現在の納期・予算に対する妥当性（見積もりレポート）」が提出されます。このレポートを「エビデンス（証拠）」として提示し、初期交渉（納期延長・予算追加）を行います。
* **エビデンスを提示して交渉する**: 顧客・上司も納得するため、初期顧客満足度の低下や上司の信頼度低下が**最小限（-5%）**に抑えられます。
* **エビデンスなしで交渉する**: 根拠がないため、顧客満足度・上司信頼度が**大幅に低下（-25%）**します。

## 📊 2. 週次スプリント監視と計算モデル

1週間（5営業日）が進む際、以下の処理が自動シミュレーションされます。

### 1. タスクの専門性とスキルミスマッチ
* すべてのタスクは `FE` または `BE` のスキルカテゴリを持ちます。
* PLは空いたメンバーに自動でタスクをアサインします。
* DEVがアサインされたタスクを作業する際、そのタスクのカテゴリとDEVの専門性が一致しない場合、以下のペナルティが発生します。
  * **作業効率**: 50% 近く低下。
  * **バグ発生率**: 通常の 3倍 に増加。

### 2. 疲労と士気の自動推移（隠しパラメータ）
* 1週間稼働したDEVは、疲労が蓄積し、士気が少し低下します。
* PMは週次のダッシュボードで、メンバーの正確な数値ではなく、PLがまとめた「良好 / 要注意 / 限界」などの大まかなサインで推移を確認します。

## 🧠 3. PMの週次戦略アクション

毎週の開始時、PMは以下のマクロコマンドを実行できます。

* **PLへの開発方針指示**: 通常（進捗優先）または バグ修正優先を指示する。
* **PLを介した要員指示 (残業・休暇)**:
  * 直接DEVに命令するのではなく、PLを通じて「来週、タクを休ませてくれ」「来週は残業でカバーしてくれ」と指示します。
  * DEVの疲労が高い状態で残業指示を強行しようとすると、PLから「これ以上の残業は避けるべきです」と進言されます。これを無視して指示を強行すると、PLのプライドが傷つき**PL士気が大幅に低下（過干渉ペナルティ: -25）**します。PL士気が0になるとボイコット（自動アサイン停止）が発生します。
* **PLとのスプリント面談**: PLの士気を回復させます。さらに、メンバー全員の正確な疲労・士気数値をレポートとして開示させます。
* **顧客との週次定例（デモ報告）**: 進捗報告とデモを行います。顧客の「要求あいまい度」を 20% 減少させ、将来の仕様変更率を下げます。
* **上司へのエスカレーション交渉**: トラブル発生時に追加予算や納期延長をエスカレーションします（上司信頼度を消費）。
* **PLへのケア（謝罪）**: PLがボイコットした際、予算を ¥50,000 消費して関係を修復し、自動アサインを復旧させます。
