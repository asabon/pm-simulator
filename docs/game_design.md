# PM Simulator ゲームデザイン仕様書 (マクロPM ➔ 要件定義・スキルセット対応版)

本ドキュメントは、「PM Simulator」のゲームルール、各種エンティティのデータ構造、およびコアループ内の計算モデルを定義します。

---

## 🎯 ゲームの基本目的と勝利条件
プレイヤーの目標は、自分をプロジェクトにアサインした**「上級マネージャー（上司）」を満足させること**です。
上級マネージャーの評価は、プロジェクトの**「最終的な顧客満足度」**、PMとしての**「適切な管理（報告・エスカレーション）」**、そして**「チームの健全性（過労離職者を出さない）」**によって決定されます。

---

## 🔁 ゲームサイクル（週次スプリント進行）

ゲームは「1週間（5営業日 / 1スプリント）」を最小の進行単位とし、キックオフから納品までをシミュレートします。

```
 [1. キックオフ（事前ヒアリング ➔ 雇用 ➔ 見積もり ➔ 事前交渉）] （初回のみ）
         │
         ▼
 [2. スプリント開始（PMの方針・ケア指示）] 
         │
         ▼
 [3. 1週間の自動シミュレーション（PLによる自動アサインと開発）]
         │
         ▼
 [4. スプリント終了（週次ダッシュボード確認）]
         │
         ▼
 [5. 週次トラブル・イベントへの対応 ＆ 顧客・上司交渉]
         │
         ▼ (クリアまたは破綻するまで 2-5 を繰り返す)
```

---

## 🤝 1. キックオフフェーズ（要件明確化・体制構築・事前交渉）

プロジェクト開始前に、PMは以下の3つのステップを経てプロジェクトの初期状態を構築・調整します。

### STEP 1: 顧客ヒアリング（要求の明確化）
顧客と対話し、要求の細部をヒアリングします。顧客タイプによってヒアリングの効果が大きく変動します。
1. **丁寧なヒアリング (納期を1週消費)**
   * **品質・スピード重視顧客**: 要件が整理され明確化されます（顧客初期満足度 +10、要求あいまい度 `vague_level` が大幅低下 -40）。
   * **要件あいまい重視顧客 (罠)**: 顧客が要求を言語化できないため、時間をかけても意味がありません（初期満足度変化なし、`vague_level` は -10 のみ低下。単に納期バッファが1週間削られるだけの罠となります）。
2. **簡易ヒアリング (即日キックオフ)**
   * 影響なし。
3. **ヒアリングなし (即日、納期+1週間)**
   * 納期を1週間余分に稼げますが、顧客を軽視したため初期満足度が低下（-15）、`vague_level` +30、初期バグ率 1.5倍。

### STEP 2: 体制構築（雇用）とPL見積もり（妥当性確認）
予算の範囲内で、雇用するPLおよびDEVを選択します。
* **PL (鈴木PL / 田中PL)**:
  * 鈴木PL（日当 ¥18,000）: 優秀。バグ混入率を15%下げる。正確な見積もりと妥当性レポートを提示。
  * 田中PL（日当 ¥10,000）: 若手。30%でアサインミス（アサイン遅延）を起こす。見積もりの精度が粗くブレる。
* **DEV (山田DEV / 佐藤DEV)**:
  * 山田 (BE特化/日当 ¥15,000): BE（バックエンド）タスク速度 1.5倍。FE（フロントエンド）タスクは苦手（速度 0.6倍、バグ率 3倍）。
  * 佐藤 (FE特化/日当 ¥12,000): FEタスク速度 1.2倍。BEタスクは苦手（速度 0.5倍、バグ率 3倍）。

メンバーを雇用した際、PLから「現在の納期・予算に対する妥当性（見積もりレポート）」が提出されます。
* 例（鈴木PL）:「総見積もり250時間です。アサインされたメンバーの合計稼働時間は200時間なので、**50時間分（約1週間分）スケジュールが不足**しています。」
* このレポートの提示を受けることで、プロジェクトが抱える「リスク（スケジュール不足、スキルミスマッチなど）」が数値で可視化されます。

### STEP 3: エビデンス（証拠）を持った事前交渉
PLの見積もりレポートを「エビデンス」として提示し、顧客や上司と初期交渉（納期延長・予算追加）を行います。
* **エビデンスを提示して交渉する**: 顧客・上司も納得するため、初期顧客満足度の低下や上司の信頼度低下が**最小限（-5%）**に抑えられます。
* **エビデンスなしで交渉する (STEP 2の見積もりをスキップした状態など)**: 根拠がないため、顧客満足度・上司信頼度が**大幅に低下（-25%）**します。

## 📊 2. 週次スプリント監視と計算モデル

1週間（5営業日）が進む際、以下の処理が自動シミュレーションされます。

### 1. タスクの専門性とスキルミスマッチ
* すべてのタスクは `FE` または `BE` のスキルカテゴリを持ちます。
* PLは空いたメンバーに自動でタスクをアサインします。
* DEVがアサインされたタスクを作業する際、そのタスクのカテゴリとDEVの専門性が一致しない場合、以下のペナルティが発生します。
  * **作業効率**: 50% 近く低下。
  * **バグ発生率**: 通常の 3倍 に増加。

### 2. 疲労と士気の自動推移（隠しパラメータ）
* 1週間稼働したDEVは、疲労が蓄積し、士気が少し低下します。
* PMは週次のダッシュボードで、メンバーの正確な数値ではなく、PLがまとめた「良好 / 要注意 / 限界」などの大まかなサインで推移を確認します。

## 🧠 3. PMの週次戦略アクション

毎週の開始時、PMは以下のマクロコマンドを実行できます。

* **PLへの開発方針指示**: 通常（進捗優先）または バグ修正優先を指示する。
* **PLを介した要員指示 (残業・休暇)**:
  * 直接DEVに命令するのではなく、PLを通じて「来週、山田を休ませてくれ」「来週は残業でカバーしてくれ」と指示します。
  * DEVの疲労が高い状態で残業指示を強行しようとすると、PLから「これ以上の残業は避けるべきです」と進言されます。これを無視して指示を強行すると、PLのプライドが傷つき**PL士気が大幅に低下（過干渉ペナルティ: -25）**します。PL士気が0になるとボイコット（自動アサイン停止）が発生します。
* **PLとのスプリント面談**: PLの士気を回復させます。さらに、メンバー全員の正確な疲労・士気数値をレポートとして開示させます。
* **顧客との週次定例（デモ報告）**: 進捗報告とデモを行います。顧客の「要求あいまい度」を 20% 減少させ、将来の仕様変更率を下げます。
* **上司へのエスカレーション交渉**: トラブル発生時に追加予算や納期延長をエスカレーションします（上司信頼度を消費）。
* **PLへのケア（謝罪）**: PLがボイコットした際、予算を ¥50,000 消費して関係を修復し、自動アサインを復旧させます。
