# PM Simulator ゲームデザイン仕様書

本ドキュメントは、「PM Simulator」のゲームルール、各種エンティティのデータ構造、およびコアループ内の計算モデルを定義します。

---

## 🎯 ゲームの基本目的と勝利条件
プレイヤーの究極の目標は、自分をプロジェクトにアサインした**「上級マネージャー」を満足させること**です。
上級マネージャーの評価は、主にプロジェクトがもたらす**「顧客満足度」**と、PMとしての**「適切な管理（報告・エスカレーション）」**、そして**「チームの維持（離職者を出さない）」**によって決定されます。

---

## 🔁 コアループと計算モデル（1日の処理）

ゲームは「1日」を最小の進行単位とします。プレイヤーが「1日を進める」を選択した際、以下の順序で処理が実行されます。

### 1. タスク進捗の更新
割り当てられている各タスクについて、担当開発者の作業が進みます。
* **1日の進捗（%）の基本計算式**:
  $$\text{進捗増加幅(\%)} = \frac{\text{開発者の作業速度} \times \text{補正係数}}{\text{タスクの見積もり時間}} \times 100$$
* **補正係数（士気と疲労の影響）**:
  $$\text{補正係数} = \left( 0.5 + 0.5 \times \frac{\text{士気}}{100} \right) \times \left( 1.0 - 0.5 \times \frac{\text{疲労}}{100} \right)$$
  *(士気が最低で作業効率50%減、疲労が最大で作業効率50%減となるモデル)*

### 2. 開発者の状態変化（隠しパラメータ）
* 開発者の「疲労」や「ストレス（士気）」は、プレイヤーから**数値としては隠蔽**されています。
* **疲労の蓄積**: タスクに従事している開発者は、1日につき **疲労が 10 増加** します（残業時は20増加）。
* **サイン（危険信号）の表出**:
  疲労やストレスが高まると、ステータス画面の「開発者のひと言」がネガティブに変化し、タスク実行時のバグ発生率が上昇します。

### 3. バグの発生
進捗が進む際、確率的にバグが発生し、プロジェクトの「バグ数」に加算されます。
* **バグ発生確率**:
  $$\text{バグ発生率} = (\text{開発者の基本バグ率} + \text{タスクの難易度によるバグ率}) \times \left( 1.0 + \frac{\text{疲労}}{100} \right)$$
  *(疲労がたまるとバグ発生率が最大で2倍に増加。プレイヤーは「バグの急増」からもメンバーの疲労を推測できます)*

### 4. ランダムイベントの判定
* 1日の終わりに、一定確率でトラブルやイベントが発生します。
* 顧客タイプが `VAGUE_REQUIREMENTS`（要件あいまい）の場合、顧客の「あいまい度 (`vague_level`)」に基づき、追加要求（再作業タスクの追加）が高確率で発生します。

### 5. プロジェクト・顧客ステータスの更新
* 顧客満足度が更新されます。顧客の重視するポイント（品質か納期か）とプロジェクトの現状に乖離があると、満足度が低下します。
* 顧客満足度が極端に低下した場合、顧客から上級マネージャーへ直接クレームが入り、上級マネージャーの満足度（評価）が大幅に低下します。

## 🏢 組織構造と権限委譲（PM - PL - DEV の3層モデル）

PMが細かい指示ばかり出して本来の業務を疎かにすることを防ぎ、階層組織での管理を学ぶため、本作は **「PM ➔ PL ➔ DEV」** の3層構造を採用します。

### 1. 自律的なタスク管理（PLの自動アサイン）
* 通常状態では、プロジェクトリーダー（PL）が、空いているメンバー（DEV）に対してTODOタスクを自動で適切な順序で割り当てます。
* PMは「どのタスクに誰を割り当てるか」の細かい指示を出す必要はありません。

### 2. PLを介した指示（方向性・方針の指示）
* PMはPLに対し、プロジェクトの開発方針を指示できます。
  * **通常方針（NORMAL）**: PLはタスクの進捗が早い順に自動アサインします。
  * **バグ修正優先方針（BUG_FIRST）**: PLはバグ数を減らすため、バグ修正タスクを最優先でメンバーに割り当てます。

### 3. マイクロマネジメント（過干渉）ペナルティ
* PMがPLを介さず、直接メンバー（山田・佐藤など）に対して「残業の指示」や「休暇の付与」といった細かい指示を出すと、**PLの士気（管理者プライド）が急低下**（1回につき -25）します。
* PLの士気が `0` になると、PLは管理をボイコットし、**自律稼働を停止**します。
* **ボイコット中の影響**:
  * 自動タスクアサインが停止します。
  * 開発者はタスクにアサインされず、何もしなくなります。
  * PMは自動的に手動での細かいタスクアサインを強制され、大量の操作やコストを強いられます。
  * PLを復職させるには、「謝罪・ケア」アクションとして予算 ¥50,000 を消費する必要があります。

### 4. PLを通じた現場レポート
* PMはPLと「1on1（面談）」を行うことで、PLが吸い上げたメンバー全員の正確なステータス（隠しパラメータである疲労度・士気の具体的な数値）を把握できます。

## 📌 データモデル（エンティティ定義）

### 1. タスク (Task)
| プロパティ名 | 型 | 説明 |
| :--- | :--- | :--- |
| `id` | String | タスクの一意な識別子 |
| `name` | String | タスク名（例: 「データベース設計」「API実装」） |
| `estimated_hours` | float | 見積もり時間（人時） |
| `actual_hours` | float | 実際に投入された累積時間 |
| `progress` | float | 進捗率（`0.0` 〜 `100.0`） |
| `assigned_developer_id` | String | 担当している開発者のID（未割り当て時は空） |
| `difficulty` | float | タスク難易度補正（バグ率や作業速度に影響） |
| `status` | Enum | `TODO` / `IN_PROGRESS` / `DONE` |

### 2. 開発者 (Developer)
| プロパティ名 | 型 | 初期値・範囲 | 説明 |
| :--- | :--- | :--- | :--- |
| `id` | String | - | 開発者の一意な識別子 |
| `name` | String | - | 開発者名 |
| `role` | String | `"PL"` / `"DEV"` | 役職（`PL`は自動アサインと全体進捗管理を行う） |
| `work_speed` | float | `0.8` 〜 `1.5` | 基本作業速度の倍率 |
| `base_bug_rate` | float | `0.01` 〜 `0.10` | 基本バグ発生率 |
| `morale` | float | `0` 〜 `100` (非表示) | 士気（高いほど作業効率向上） |
| `fatigue` | float | `0` 〜 `100` (非表示) | 疲労（高いほど効率低下、バグ率上昇） |
| `personality_tags` | Array[String] | - | 隠し属性タグ |
| `salary` | int | - | 1日あたりの雇用コスト |

### 3. 顧客 (Customer)
| プロパティ名 | 型 | 説明 |
| :--- | :--- | :--- |
| `id` | String | 顧客の一意な識別子 |
| `name` | String | 会社名・担当者名 |
| `type` | Enum | `SPEED_ORIENTED`（納期優先） / `QUALITY_ORIENTED`（品質優先） / `VAGUE_REQUIREMENTS`（要件あいまい） |
| `satisfaction` | float | 顧客満足度（`0` 〜 `100`） |
| `vague_level` | float | 顧客の要求あいまい度（`0` 〜 `100`） |
| `report_deficit` | float | 未報告のバグや遅延による「隠れたリスク値」 |

### 4. プロジェクト (Project)
| プロパティ名 | 型 | 説明 |
| :--- | :--- | :--- |
| `name` | String | プロジェクト名 |
| `budget` | int | 残予算 |
| `deadline_days` | int | 納期までの残り日数 |
| `bugs_total` | int | 検出されている未解決のバグ数 |
| `customer_id` | String | 担当顧客のID |
| `pl_active` | bool | PLが自律稼働しているか（Trueの時、タスクは自動割り当てされる） |
| `pl_morale` | float | PLの士気・管理者プライド（`0` 〜 `100`。0になるとボイコット） |
| `direction` | String | 現在の開発方針（`"NORMAL"`: 進捗優先 / `"BUG_FIRST"`: バグ修正優先） |

## ⚠️ 報告（エスカレーション）と「こっそり解決」のジレンマ

プロジェクトで遅延やバグ多発などの問題が発生した際、PMには以下の2つのルートが提供されます。

### A. 正指に報告する（エスカレーション）
* **行動**: 上級マネージャー、または顧客へ早期にトラブルを報告する。
* **結果**: 短期的には「管理能力不足」として評価が少し下がる、あるいは顧客満足度が一時的に低下する。しかし、上司から「応援要員の追加」などの支援措置を得られたり、顧客から納期延長の合意を取ったりできるため、致命的な「炎上（即ゲームオーバー）」を確実に回避できます。

### B. 報告せず「こっそり解決」を試みる
* **行動**: 問題を報告せず、現場のメンバーに「残業」を指示したり、裏で仕様を削ったりして帳尻を合わせようとする。
* **成功時**: 顧客や上司にバレずに期日通りに解決できた場合、ペナルティは一切なく「完璧なPM」として高い評価を維持できます。
* **失敗時**: 解決できずにメンバーが過労で退職したり、納品時にバグや遅延が発覚（サプライズ）した場合、顧客満足度は壊滅し、上級マネージャーの評価も最低（即ゲームオーバー）になります。

## 🛡 顧客からの追加要望とPMの「防波堤（クッション）」機能

顧客タイプが `VAGUE_REQUIREMENTS`（要件あいまい）の場合、一定のあいまい度（`vague_level`）に応じて「追加要求（手戻りタスクの発生）」イベントが発生します。
追加要求が発生した際、PMは以下の3つの選択から対応を決定します。

1. **丸投げする**
   - **内容**: 顧客の要望をそのまま開発メンバーに突き落とす。
   - **結果**: 顧客満足度は大きく上昇するが、担当となる開発者の「士気」が激減（-30）し、ストレスが増加する。
2. **防波堤（クッション）として調整する (APを消費)**
   - **内容**: PMが自ら調整・説明の責任を負い、開発者に納得感を持って仕様修正を依頼する。
   - **結果**: 行動ポイント（AP）を消費し、顧客満足度の上昇幅はやや小さくなるが、開発者の士気低下を最小限（-5）に抑えることができる。
3. **顧客と交渉して断る**
   - **内容**: 現実的なスケジュールとメンバーの負荷を盾に、追加要求を突っぱねる。
   - **結果**: 追加タスクは発生しないが、顧客満足度が大幅に低下する。

## 🧠 PMの日常アクション（能動的アプローチ）

待っているだけではプロジェクトは沈むため、PMは手持ちの行動ポイント（AP）や予算を使い、以下のコマンドを実行する必要があります。

* **1on1（面談）**: PL、または開発者と対話する。
  * **PLと1on1を行う**: PLの士気が大幅に回復（ボイコット防止）。さらに、PLから現場メンバー（山田・佐藤）の正確な「疲労」「士気」の数値データ（レポート）が提供される。
  * **DEVと直接1on1を行う**: 開発者個人のステータスが開示されるが、PLから「現場へのマイクロ過干渉」と受け取られ、**PLの士気が少し下がる（-10）**。
* **飲み会（懇親会）**: 予算を消費して懇親会を企画する。
  * `DRINK_LOVER`（飲み会好き）のメンバー ➔ 士気が大幅アップ。
  * `PRIVATE_FIRST`（残業嫌い）のメンバー ➔ 自由参加なら影響なし。強制参加させると士気が大幅ダウン。
* **方針指示（PL指示）**: PLにプロジェクトの優先方針（NORMAL / BUG_FIRST）を指示する。
* **顧客への進捗報告・期待値調整（デモ）**: 顧客にデモを見せたり、現状を説明する。
  * 顧客満足度の低下を防ぐ。
  * `VAGUE_REQUIREMENTS` 顧客の場合、**あいまい度 (`vague_level`) を 20% 引き下げる**効果があり、将来の仕様変更や手戻りタスクの発生率を低減できる。
* **現場指示（残業・休暇）**: PLのボイコット中、または過干渉を承知で直接メンバーに指示を出す。
  * 直接山田や佐藤に残業・休暇を命令すると、**PLの士気が急低下（-25）**する。
* **PLへの謝罪・ケア**: PLがボイコットした際、予算 ¥50,000 を消費して謝罪し、PLの士気を 50 まで戻して自律アサインを復旧させる。

## 📈 将来の拡張構想（スケールアップ）
今回はプロトタイプ（MVP）としてPLを1名とする3層構造で進めますが、将来的な中〜大規模プロジェクトシナリオでは以下の拡張を行います。
* **複数チームとチームリーダー（TL）の導入**:
  * 開発メンバーが8〜10名に増加し、「フロントエンドチーム」「バックエンドチーム」などに分割される。
  * 各チームに「TL (Team Leader)」が配置され、PLが各TLを取りまとめ、PMはPLから報告を受ける、という多層階層モデルによるスケールアウトマネジメントをシミュレートします。
