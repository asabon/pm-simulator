# PM Simulator - AI開発ガイドライン & ルール

本ドキュメントは、Google Antigravity が本リポジトリで作業する際に遵守すべき開発ルールとコンテキストを定義します。

---

## 🎯 プロジェクトの文脈
* **ゲーム概要**: プレイヤーが責任感を持って動き、ステークホルダーやチームをマネジメントする「プロジェクトマネージャー体験シミュレーションゲーム」
* **開発アプローチ**:
  1. まずは `prototype/` 内の Python CLI版でゲームバランスとジレンマの面白さを検証する。
  2. ロジックが完成し合意が取れたら、Godot 4 (GDScript) による本格的な開発・UI実装に移行する。

---

## 🛠 AI支援開発の原則

### 1. 議論と仕様化の優先
* 実装に入る前に、必ずユーザーと議論を重ねて仕様を徐々に具体化・明文化すること。
* 大きな変更や新機能の追加時は、まず実装計画（Implementation Plan）を提示してユーザーの承認を得るプロセスを徹底すること。

### 2. ドキュメントの整合性
* 仕様に関する合意が得られたら、実装前に `docs/game_design.md` を必ず最新の状態に更新すること。コードと仕様書の間に乖離を作らないこと。

### 3. 技術的ルール
* **Python (プロトタイプ)**:
  * ターミナル上で対話的に動作し、テストプレイが容易なシンプルなコードを維持する。
  * オブジェクト指向の明確な責務分け（データモデルとゲームエンジン）を意識する。
* **Godot 4 / GDScript (将来の移植時)**:
  * 静的型ヒント（型アノテーション）を徹底する（例: `var score: int = 0`, `func update() -> void`）。
  * UI構築の際は Control ノードのアンカーとコンテナ（HBox/VBox/Grid）を正しく設定し、レスポンシブなレイアウトにすること。
  * セーブデータのシリアライズが容易な、辞書（Dictionary）変換可能なデータクラス設計を行うこと。

### 4. Git・ブランチ管理のルール
* `main` ブランチへの直接のコミットおよびプッシュは**厳禁**とする。
* 新機能の開発やリファクタリング、ドキュメントの更新などすべての作業は、必ず `main` ブランチから切ったフィーチャーブランチ（例: `feature/xxx`）で行うこと。
* コミット後は必ずリモートにプッシュし、GitHub CLI (`gh`) が使用可能であれば自動でプルリクエストを作成すること。`gh` が未認証等の理由でエラーになる場合は、プッシュ時に得られるプルリクエスト作成URLをユーザーに提示し、マージ要求を明示すること。

---

## 📂 フォルダ構成のルール
* ドキュメントは `docs/` 配下に格納する。
* プロトタイプは `prototype/` 配下に格納する。
* Godot関連のスクリプトは将来的に `scenes/` または `src/` 配下に構造化して配置する。
