# PM Simulator - AI開発ガイドライン & ルール

本ドキュメントは、Google Antigravity が本リポジトリで作業する際に遵守すべき開発ルールとコンテキストを定義します。

---

## 🎯 プロジェクトの文脈
* **ゲーム概要**: プレイヤーが責任感を持って動き、ステークホルダーやチームをマネジメントする「プロジェクトマネージャー体験シミュレーションゲーム」
* **開発アプローチ**:
  1. **仕様化**: `docs/` 以下で、ゲームのコンセプトや仕様を文書化する。
  2. **プロトタイピング**: `prototype/` 内の Python CLI 版で、ゲームバランスとジレンマの面白さ、簡単な動作確認を行う。
     * ※ 迅速な原理確認のため、1 の「仕様化」と 2 の「プロトタイピング」は並行して進めてもよい。
  3. **本格開発**: 仕様やロジックが固まり合意が取れたら、Godot 4 (GDScript) による本格的な開発・UI実装に移行する。

---

## 🛠 AI支援開発の原則

### 1. 議論と仕様化の優先
* 実装に入る前に、必ずユーザーと議論を重ねて仕様を徐々に具体化・明文化すること。
* 大きな変更や新機能の追加時は、まず実装計画（Implementation Plan）を提示してユーザーの承認を得るプロセスを徹底すること。

### 2. ドキュメントの整合性
* 仕様に関する合意が得られたら、実装前に `docs/game_design.md` を必ず最新の状態に更新すること。コードと仕様書の間に乖離を作らないこと。

### 3. 技術的ルール
* **コード記述時のコア要件 (実装時必須)**:
  * **Python (プロトタイプ)**:
    * エントリポイントで標準出力をUTF-8に再設定（`sys.stdout.reconfigure(encoding='utf-8')` など）すること。
    * オブジェクト指向の明確な責務分け（データモデルとゲームエンジン）を意識すること。
  * **Godot 4 / GDScript (将来の移植時)**:
    * 静的型ヒント（型アノテーション）を徹底すること（例: `var score: int = 0`, `func update() -> void`）。
    * Controlノードのアンカーとコンテナ（HBox/VBox/Grid）を正しく設定し、レスポンシブなレイアウトにすること。
    * 辞書（Dictionary）変換可能なセーブデータシリアライズ設計を行うこと。
  * ※ PR発行前の詳細な品質検証項目・チェック手順については `.agents/skills/code-review/SKILL.md` を参照すること。

### 4. Git・ブランチ管理 ＆ PR自動化ルール
* `main` ブランチへの直接のコミットおよびプッシュは**厳禁**とする。
* 新機能の開発やリファクタリング、ドキュメントの更新などすべての作業は、必ず `main` ブランチから切ったフィーチャーブランチ（例: `feature/xxx`）で行うこと。
* **PR作成フロー (`create-pr` スキルの実行)**:
  * ユーザーから PR の作成を依頼された場合（例: 「PR出してください」「PR作成」等）は、必ず `.agents/skills/create-pr/SKILL.md` のワークフローに従うこと。
  * スキル実行時に専門スキル（`code-review`, `doc-review`, `run-tests`）による自動検証・自動修正を行い、統一PRテンプレート（目的・概要・検証/自動修正結果）を適用して GitHub CLI (`gh`) からPRを発行すること。

## 📂 フォルダ構成のルール
* ドキュメントは `docs/` 配下に格納する。
* プロトタイプは `prototype/` 配下に格納する。
* カスタムスキルは `.agents/skills/<skill_name>/SKILL.md` に定義する。
* Godot関連のスクリプトは将来的に `scenes/` または `src/` 配下に構造化して配置する。
